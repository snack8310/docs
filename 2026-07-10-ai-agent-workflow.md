# 分享记录：AI Agent 协作工作模式（AGENTS.md 现行标准实践）

日期：2026-07-10

## 背景

来自 `bsp` 项目 `AGENTS.md` 的实践：把团队协作规则、历史记忆、需求证据，都沉淀成 AI 能直接读懂并遵循的结构化文档，让"人机协作的工作方式"本身变成可版本管理、可迭代的资产，而不是散落在脑子里或聊天记录里。

核心思路：**把工作模式逐步转化为 AI 看得懂的描述流程**——不是写给人看的规范，而是写给 Agent 执行时会真正读取、真正遵守的"协议文档"。

## 1. Agent 文档先行，定标准

`AGENTS.md` 作为仓库共享真源（`CLAUDE.md`、`.claude/` 等工具私有文件不算数），先于具体任务定义好边界：

- 全局约束：语言规范（响应用中文、代码注释用英文等）、禁止直接下发生产变更、禁止索要敏感信息、信息不足时列清单而非猜测。
- 规则优先级：平台安全合规 > 用户当前会话指令 > 根目录 AGENTS.md > 其他参考文档；子目录 AGENTS.md 只影响子树，遵循"全局先满足、局部再细化"。

这样不同 Agent、不同会话进入同一个仓库时，行为基线是一致的，不依赖每次临时交代。

## 2. 主 Agent + 评估器双环

关键设计：**实施者和裁判物理隔离**。

- 主 Agent 负责读仓库、改代码、补测试、跑验证、生成交付说明，但**不得自行宣布完成**。
- 评估器是独立子 Agent/独立会话，只拿"任务卡 + 实现代码/diff"，不携带主 Agent 的心路历程；只读权限，不能改代码；裁决不可被主 Agent 覆盖。
- 评估器的方法论是对抗性的：默认假设有问题，主动找反例和边界条件，找不到问题才判定通过——不是逐项打勾。

这一环节把"自己给自己批作业"的风险结构性地消除了。

## 3. 第一性原理贯穿三个环节

- 任务卡阶段：拆解表面需求，识别真实问题和硬约束。
- 方案设计阶段：验证有没有更本质的解法，不直接套用惯例。
- 评估器审查阶段：核对是否真正解决了任务卡里的本质问题，而不是"看起来对不对"。

## 4. 任务卡：把自然语言收敛成结构化输入

日常输入允许自然语言，但**进入实施前必须整理成任务卡**，未完成整理不进入代码修改阶段。

- 小任务：标题、类型、背景、目标、范围、验收标准、风险点。
- 大任务：在小任务基础上再加非目标、影响面、验证方案。

按小/大任务分级治理执行规则（自修复轮数上限、是否需要人审）也不同，避免一刀切。高风险操作（数据库结构变更、支付链路、幂等/防重/权限等关键逻辑）额外设卡点，必须人工确认。

## 5. 记忆分层：变更证据 vs 抽象记忆

这是"生成需求证据"的具体落地方式，分两层，各司其职：

- `docs/history/changes/`：**变更证据**，单次需求/缺陷/重构的记录——背景、关键实现、关键文件、验证结果、风险与未覆盖项。写事实，不写长报告。
- `docs/history/memory/`：**抽象记忆**，长期有效的业务规则、实现约定、关键权衡、踩过的坑。写稳定规则，不写过程流水账。评估器审查通过后，由评估器（独立视角）判断本次任务是否值得沉淀成抽象记忆。

配套的还有 `docs/storybook.md`（用户故事清单）和 `docs/testing/baseline.md`（测试基线，定义"这次改动至少要证明什么没有坏"）。

## 6. AI 建立上下文的默认顺序

```
AGENTS.md → README.md → docs/history/memory/ → docs/storybook.md
→ 当前相关 specs/ 或功能文档 → docs/history/changes/ → 相关代码与测试
```

这个顺序本身就是"先规则、再长期记忆、再需求清单、再具体任务上下文"的心智模型，避免 Agent 一上来就扎进代码细节。

## 方法论总结

1. **协作规则显式化**：把"怎么和 AI 一起干活"写成仓库里的一等公民文档（AGENTS.md），而不是隐性默契。
2. **执行与裁决分离**：双 Agent 双环，评估器对抗性审查，杜绝自证清白。
3. **需求先结构化，再进代码**：任务卡是从自然语言到可执行任务的强制关卡。
4. **证据分层沉淀**：变更证据记事实（短期），抽象记忆记规则（长期），二者不混用。
5. **分级治理**：任务大小、风险高低决定审核轮次和人工介入点，不搞一刀切的重流程。

## 待推广的点

- 这套模式目前在 `bsp` 项目落地，值得作为团队通用的 AI 协作标准模板，推广到其他仓库时按需裁剪（如去掉本项目特有的 Go 技术栈约定）。
- "评估器与主 Agent 隔离"这个设计是防止"AI 自己骗自己"的关键机制，值得重点讲给团队听。
