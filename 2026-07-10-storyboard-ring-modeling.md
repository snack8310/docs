# 分享记录：用户故事驱动的领域建模（story-board 项目模式）

日期：2026-07-10

## 背景

来自 `story-board` 项目的实践：不先画架构图，而是先把需求写成一条条用户故事，再从故事里"长出"领域模型和分层结构。核心动作是"讨论"——多人围绕故事反复对齐颗粒度，逐步收敛出准确、可测试的需求描述。

## 1. 从用户故事开始

用户故事遵循黄金三段式：

> **作为一个** [某种类型的用户/角色]，**我希望** [执行某个操作/实现某个功能]，**以便于** [获得某种商业价值/解决某个痛点]。

story-board 项目里落地为统一的 issue 模板：

```
**<角色>** 想要 _<诉求概述>_

## 谁
<角色详细描述>

## 诉求
<具体诉求>

## 价值
<带来的业务价值>
```

一个好故事要满足 **INVEST** 原则：Independent（独立）、Negotiable（可协商）、Valuable（有价值）、Estimatable（可估算）、Small（短小）、Testable（可测试）。

用户故事的完整性由 **3C** 保证：
- **Card**：卡片本身，点到为止
- **Conversation**：围绕卡片的讨论，明确细节、边界、可行性
- **Confirmation**：验收条件（Given-When-Then），判断"做完了"的标准

关键认知：**用户故事既是需求文档，也是测试用例**——验收条件天然就是测试断言，讨论过程本身就是在提炼测试场景。

## 2. 划分业务域

先粗略识别核心用户/角色，例如信贷场景里的：客户（借钱的人）、资方（投钱的人）、平台（撮合的人）。这只是讨论的起点，不是结论，后续随着故事积累会调整。

再把散落的故事归拢到业务域（如 Identity & Profile、借贷域、资金流转域），业务域下再抽象出领域模型（类图）。

## 3. 分层：Ring0 / Ring1 / Ring2

这是本次实践里最有价值的部分——用"环"来区分模型的稳定性：

- **Ring0**：核心链路的领域模型。最稳定，代表业务不可动摇的核心概念和主流程（如身份认证、授信、放款、还款）。
- **Ring1**：围绕 Ring0 的**能力/决策层**增强。是"平台需要具备什么能力"，相对稳定，但比 Ring0 更容易随业务演进。
- **Ring2**：实现 Ring1 能力的**具体技术手段/实现细节**，最容易变动（具体算法、运营战术、技术选型、单点功能）。

判断一条故事该归入哪一环，靠的是反复追问"这是能力本身，还是实现这个能力的手段？"。例如：

- "风控想要拉黑客户" → Ring1（决策能力）
- "识别撞库、多头、黑名单等欺诈风险" → Ring2（具体识别手段，会变）
- "客户完成证件OCR、活体识别" → 排除，属于 Ring0 的实现细节，不算增强能力

这个判断没有标准答案，需要多人讨论、来回调整（story-board 里能看到大量"讨论后下沉为 Ring2""讨论后排除"的记录），最终收敛出一份高置信度的 Ring1 清单。

### Ring1 起，强调数据的可观测性

从 Ring1 开始，每条故事在讨论时都要多问一句：**这个能力/决策，靠什么数据判断"做得对不对、做得好不好"？**

- Ring0 关心"事情本身有没有发生"（如：是否完成放款），Ring1 关心的是"能力/决策的效果如何"（如：风控策略拦截是否精准、授信决策是否合理），这类问题天然需要可观测的数据支撑，否则能力本身无法被验证、调优、追责。
- 具体到故事上，Ring1 的验收条件里应该显式包含：这个决策/能力对应的**关键指标**是什么、**日志/埋点**记录了哪些字段、出了问题能不能**追溯**到具体的判断依据（例如 story-board 里"配置、灰度、实验和监控风控策略效果"本身就是一条 Ring1 故事，核验的正是可观测性）。
- 一条 Ring1 故事如果说不清楚"这个能力靠什么数据验证效果"，往往说明它还停留在 Ring2 的具体手段层面，或者故事本身还不够清晰，应该回到讨论环节继续拆解。
- 可观测性要求也会反过来约束 Ring2 的实现：具体技术手段可以变，但产出的可观测数据（指标、日志字段、审计记录）的语义应尽量保持稳定，这样 Ring1 层的验收标准才不会随 Ring2 实现变动而失效。

## 4. 方法论总结

1. 不追求一次性画对架构，先让故事说话。
2. 用 INVEST + 3C 保证故事本身的质量。
3. 用 Ring0/1/2 的分层判断，把"核心稳定的"和"容易变的"分开管理，避免过度设计或过早固化。
4. 从 Ring1 开始，验收标准里必须体现数据可观测性——能力/决策的效果要有指标、日志、可追溯的判断依据，否则不算做完。
5. 全程靠多人对话收敛，讨论记录本身要沉淀下来（如 `ring-classification-notes.md`），保留判断依据，而不仅仅是结论。
6. 最终产出的用户故事集合，同时服务于需求对齐和测试验收两个目的。

## 待推广的点

- 这套"故事 → 领域划分 → Ring 分层"的模式目前在 story-board 项目验证过，值得在其他业务域复用。
- 讨论沉淀（如 ring-classification-notes.md）证明了"过程记录"和"最终结论"同样重要，避免团队日后对同一条故事反复重新讨论。
